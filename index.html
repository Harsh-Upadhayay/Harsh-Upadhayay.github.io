Self-Hosted Infrastructure
Homelab Server
neovara.uk  ·  Production
A production-grade, self-hosted infrastructure running 15+ services on a personal server — designed, built, and operated end-to-end. The stack mirrors real-world cloud architecture: zero-trust network access, SSO across all services, group-based RBAC, full observability, and automated deployments.

  • Watchtower for automated image pulls and zero-downtime container updates with health check gating
  • All stacks managed via a Makefile-driven workflow with shared env layout: global secrets in ops/.env.local (git-ignored), per-app non-secrets committed in app.env
  • Security hardening: no-new-privileges on all containers, read-only filesystems where possible, scoped Docker network isolation

03
Distributed Systems — Go
Rate Limiter
github.com/Harsh-Upadhayay/ratelimiter
A production-architected distributed rate limiting library in Go, built test-first with a deep focus on distributed systems design — interface-based algorithm abstraction, CAS-based concurrency, sharded lock striping, and a full suite of Architecture Decision Records.
  • Pure functional algorithm design: every algorithm implements Decide(now, state) → (result, newState) — no side effects, no I/O, no time.Now(); enables deterministic testing and hot-swap of backends
  • StateStore interface separates storage from logic: Get(key) + CompareAndSwap(key, version, state); missing key = version 0; CAS conflict = ok:false, err:nil
  • Algorithms implemented: Fixed Window and Token Bucket (lazy refill, float64 tokens, clock clamp for backward movement)
  • CAS retry loop (bounded to 10 attempts → ErrCASConflict): every retry re-evaluates Decide, not just retries the write — correct under concurrent mutation
  • ShardedMemoryStore: lock striping via FNV hash across N shards (configurable); benchmarks confirmed single-mutex MemoryStore serialises all keys under parallel load; sharding restores throughput
  • 57 Architecture Decision Records documenting every trade-off (Mutex vs Channels, Lua vs native Redis commands, per-key locks vs sharding, injectable clock design)
  • Planned: RedisStore backend (Lua scripts + Sorted Sets for sliding window log), AWS deployment (ECS Fargate + ElastiCache), OpenTelemetry metrics, k6 load testing
